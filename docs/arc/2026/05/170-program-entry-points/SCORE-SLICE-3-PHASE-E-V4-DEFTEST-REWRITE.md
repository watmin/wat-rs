## Root cause — prelude semantic shift breaks callers using prelude as sandbox-internal code

### What was attempted

The V4 rewrite produced the BRIEF's target expansion:

```scheme
(:wat::core::do
  ~@prelude
  (:wat::core::define (~name -> :wat::kernel::RunResult)
    (:wat::test::run-hermetic ~body)))
```

Written to `wat/test.wat`; workspace run produced 16 failures across two failure patterns.

### Failure pattern 1 — unresolved struct/enum accessors at startup

File: `wat-tests/service-template.wat`, `wat-tests/edn/roundtrip.wat`

Error:
```
test-runner: .../service-template.wat: startup: resolve: 61 unresolved reference(s):
  - :svc::State/new (call head — not a builtin, not a registered function)
  - :svc::Request::Push (call head — not a builtin, not a registered function)
  [... 59 more ...]
```

Root cause: These test files use `make-deftest` with preludes containing:
1. `(:wat::core::struct :svc::State ...)` — a struct declaration
2. `(:wat::core::enum :svc::Request ...)` — an enum declaration
3. Helper `define` forms that call `:svc::State/new`, `:svc::Request::Push` etc. — constructors/accessors generated by struct/enum at EVAL time

Under the OLD expansion, the prelude was inside `(:wat::core::forms ...)` — quoted AST data passed to `run-sandboxed-ast`. The outer resolver never sees the inner forms as live code.

Under the NEW expansion, the prelude forms are top-level in the `do` wrapper. The `do` form goes into `rest` after `register_defines`. When `resolve_references` (step 7) runs on `rest`, it recurses into the `do` body and sees the helper `define` bodies calling `:svc::State/new` etc. But struct/enum declarations are evaluated LATER (at runtime, not at `register_defines` time). The generated constructors/accessors are not in `sym.functions` at resolve time. Every such call head is `UnresolvedReference`.

`preregister_fn_defs_in_do` only pre-registers `def`-shape and `define`-shape functions — it does NOT process struct/enum declarations to pre-generate their constructors/accessors. There is no hook that pre-generates struct accessors from a `struct` form inside a `do` at resolve time.

### Failure pattern 2 — nested `:user::main` seen as call head at startup

File: `wat-tests/kernel/services/ambient-stdio.wat`

Error:
```
test-runner: .../ambient-stdio.wat: startup: resolve: 25 unresolved reference(s):
  - :user::main (call head — not a builtin, not a registered function)
  [... 24 more ...]
```

Root cause: This file uses `make-deftest-hermetic` with preludes containing helper `define` forms that call `(:wat::test::run-hermetic-ast (:wat::test::program (:wat::core::define (:user::main -> ...) ...)) ...)`.

`(:wat::test::program ...)` is a macro that expands to `(:wat::core::forms ...)` — a variadic-quote substrate form that captures its arguments as AST data (Vector<WatAST>). The `define (:user::main ...)` form inside `forms` is data, not live code.

Under the OLD expansion, these helper defines were inside the sandboxed program's AST — the outer resolver never saw them. Under the NEW expansion, they are top-level in the `do` form. The outer `resolve_references` recurses into the helper `define` bodies. It encounters `(:wat::test::run-hermetic-ast (:wat::test::program ...))`. `program` is a macro — at type-check/resolve time, macros may be expanded or their arguments may be analyzed. The `forms` arguments include `(:wat::core::define (:user::main -> ...) ...)` where `:user::main` appears as a call head in a nested form inside `program`'s AST argument.

The exact mechanism: `resolve_references` descends into the argument of `program`/`forms` and finds `:user::main` as a call head. `:user::main` is not in `sym.functions`. `UnresolvedReference` is emitted.

### Failure pattern 3 — type registry missing in hermetic child (edn::read)

File: `wat-tests/edn/roundtrip.wat`

Error:
```
thread '...' panicked at src/test_runner.rs:487:17:
test roundtrip.wat :: wat-tests::edn::roundtrip-enum-variant
  failure: forked program exited 1
```

Root cause: The prelude in `roundtrip.wat` defines `(:wat::core::enum :test::Event ...)` and `(:wat::core::struct :test::Wrapper<E> ...)`. Under the NEW expansion, these are declared in the OUTER frozen world. When `(:wat::test::run-hermetic <body>)` spawns a child via `spawn-process`, the child's world is built from `extract_closure`'s prologue. `extract_closure` captures only types that appear in the fn signature or captured values — not all types from the parent world. The child subprocess does not have `test::Event` or `test::Wrapper<E>` in its type registry. The body calls `(:wat::edn::read s)` which attempts to deserialize a tagged EDN form (`#test/Event ...`) via the type registry lookup. The lookup fails (type not registered in child), causing exit code 1.

### Why Gap E probes passed but Phase E V4 failed

Gap E probes (`probe_do_define_two_vars_visible`, `probe_do_define_via_macro_emission`) use pure `define` forms whose bodies reference only other `define` forms from the same `do`. No struct/enum accessors, no nested programs, no `forms`-quoted AST, no child subprocess type registry. Gap E proved that `define`-inside-`do` pre-registers correctly — which is true. The failure is at a higher semantic layer.

### The architectural gap (Gap F)

The BRIEF V3's framing: "preludes are usually `:wat::load-file!`, `:wat::core::defmacro`, type declarations: top-level forms that need to register at FREEZE time." This framing was aspirational — it described the INTENDED use of preludes. The actual callers in the codebase use preludes as "sandbox-internal initialization code":

1. **Struct/enum declarations + accessors**: intended to be visible inside the sandbox's runtime, not as live-call-head-resolvable forms in the outer world.
2. **Helper defines calling `run-hermetic-ast`/`program`**: intended to be inner sandbox helpers that spin up sub-programs; contain `:user::main` as inner program entry points.
3. **Type registrations needed by child subprocess**: types declared in prelude must be visible to the `edn::read` call inside the hermetic child, not just in the outer frozen world.

The prelude semantic shift (prelude at expansion site vs prelude inside sandbox) is correct in ARCHITECTURE but conflicts with the EXISTING CALLERS' prelude semantics. Migrating to the new shape requires either:

- **Call-site sweep** (prohibited by BRIEF): modify each prelude to be "freeze-time-only" forms (loads, defmacros only); move struct/enum/helpers into the test body or separate files.
- **New substrate capability (Gap F)**: a mechanism that makes struct/enum inside a top-level `do` pre-generate their accessors at resolve time (so the outer resolver can validate calls to those accessors).
- **Alternative expansion shape**: the `do` splice ONLY for freeze-time forms (loads, defmacros) while struct/enum/program-helpers remain inside a `forms` wrapper that is passed through to the sandbox.

None of these are in Phase E V4's scope without violating constraints.

### Delta 4 — The call-site preludes are "sandbox-internal" not "freeze-time"

The V3 BRIEF stated preludes are "usually `:wat::load-file!`, `:wat::core::defmacro`, type declarations: top-level forms that need to register at FREEZE time." Inspecting the actual callers:

- `wat-tests/edn/roundtrip.wat`: prelude contains `enum` + `struct` declarations — used inside the sandbox at runtime
- `wat-tests/service-template.wat`: prelude contains `struct` + `enum` + helper `define` forms with accessor calls — all sandbox-internal
- `wat-tests/kernel/services/ambient-stdio.wat`: prelude contains `define` forms with nested `program`/`:user::main` — sandbox-internal helper programs
- `wat-tests/holon/ReciprocalLog.wat`: prelude is `()` — safe under new expansion

3/5 make-deftest files use preludes that are semantically sandbox-internal, not freeze-time. The new expansion shape requires a different prelude taxonomy.

### Delta 5 — struct/enum-in-do is a deeper substrate gap than define-in-do

For the new expansion to work with struct/enum in prelude, `preregister_fn_defs_in_do` would need to also call `startup_from_source`-style struct/enum processing at pre-registration time — generating accessor stubs for `sym.functions`. This is a non-trivial substrate arc (Gap F candidate). It would need to handle all struct/enum accessor name patterns, generic types, and potential ordering dependencies between struct definitions.

## Substrate gap needed (Gap F) — struct/enum accessors in top-level `do`

**One or more substrate changes are needed to unblock Phase E V5:**

**Gap F-A (strict prerequisite)**: Extend `preregister_fn_defs_in_do` to also handle `(:wat::core::struct ...)` and `(:wat::core::enum ...)` forms by pre-generating their accessor/constructor names into `sym.functions`. This requires extracting the name-generation logic from the struct/enum eval path and making it callable at resolve time (before eval). This is non-trivial: struct fields, enum variants, generic parameters all contribute generated names.

**Gap F-B (alternative)**: Rather than pre-generating accessors, maintain an "unresolved-ok" set for call heads that are known to be generated by struct/enum declarations in the same `do` body. The resolver would skip validation for these call heads, deferring to eval time. This is simpler but weaker (hides real resolve errors).

**Gap F-C (architectural alternative)**: Introduce a prelude taxonomy split — `load-prelude` (freeze-time: loads, defmacros, load-file!) vs `sandbox-prelude` (sandbox-internal: struct/enum/program-helpers). Deftest takes two prelude slots. Callers with sandbox-internal preludes pass them to `sandbox-prelude`. This is a call-site-visible change but an honest one.

**Gap F-D (pragmatic)**: Sweep the existing callers (3 files) to move sandbox-internal prelude content into separate load files or into the test bodies directly. The make-deftest prelude becomes freeze-time-only. This is the call-site change the BRIEF prohibits for Phase E V4 — but may be the right arc for V5 with orchestrator approval.

**Recommendation for orchestrator**: Gap F-D (prelude sweep) is the most honest and structurally clean path. The existing callers are using prelude semantically wrong (sandbox-internal code is not freeze-time code). Correcting the call sites is honest; adding substrate complexity to paper over the semantic mismatch is not. A Gap F-D sweep arc would:
1. Move `struct`/`enum`/`define`-with-accessors from preludes into either load files (`:wat::load-file!` in the prelude) or into test file preambles
2. Move nested-program helpers out of preludes into separate helper files
3. Leave preludes as `()` or pure load-file! preludes
Then Phase E V5 ships with the new expansion shape and all callers pass.
