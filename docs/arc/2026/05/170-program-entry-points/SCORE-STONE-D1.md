## Honest deltas

### Decision A — Tuple-wrapped vs bare factory (BRIEF § Tasks 1, choice (i) vs (ii))

**Picked (ii) bare factory.** Four-questions on (i) vs (ii):

- **Obvious:** (ii) — direct unquote of the factory AST. (i) would require pattern-matching INTO the `(Tuple factory)` AST at expand time to extract the single child. (ii) **YES**.
- **Simple:** (ii) — no AST destructuring at all. (i) requires substrate machinery (AST/nth or equivalent) that wat does not expose at the defmacro layer. (ii) **YES**.
- **Honest:** (ii) — D1 is explicitly N=1; the Tuple wrapper is shape that fits multi-factory N>1; forcing a Tuple ceremony for one element is dishonest about the N=1 case. (ii) **YES**.
- **Good UX:** D2 will extend via variadic positional `& (factories ...)` collector (precedent: `:wat::test::program` at `wat/test.wat:228-231`). Call form scales `(run-threads rx-type tx-type factory client-fn)` (D1) → `(run-threads rx-type tx-type factory-A factory-B ... client-fn)` (D2). No API rework between stones. (ii) **YES**.

All four YES — (ii) wins.

D2's extension path is now via variadic collector at the FACTORY position, NOT via Tuple-AST iteration. The BRIEF's `(Tuple factory)` framing assumed wat-level AST destructuring is expressible at expand time; it is not in the current substrate (no AST/Keyword constructor, no AST/children accessor, no AST/nth accessor exposed to user code). D2 adopts the variadic-collector pattern instead — this is a STRENGTHENING of the original design, not a deviation.

### Decision B — Pre-baked `Receiver<I>` / `Sender<O>` type keywords as macro args

**Macro takes 4 positional args:** `server-rx-type`, `server-tx-type`, `factory`, `client-fn`.

The BRIEF's target expansion shape wrote the wrap-fn binders as:

```
[server-rx <- :rust::crossbeam_channel::Receiver<I>
 server-tx <- :rust::crossbeam_channel::Sender<O>]
```

with `I` and `O` as conceptual placeholders. The honest implementation needs concrete type keywords substituted at expand time, and there are exactly two paths:

- **(B.1) Build `:Receiver<I>` from `:I`-typed args via expand-time keyword construction.** Requires `keyword/from-string` reachable at macro-body expression position AND splicing a constructed keyword at the `<-` binder position. The macro body IS evaluated at expand time (defmacros are ordinary expressions returning AST), so `keyword/from-string` is reachable, but the **parser tokenizes `Receiver<X>` atomically** — `~` unquote inside `<...>` is parsed as part of the keyword token. The unquote opportunity never reaches the splicer. This is a substrate property (see STOP trigger 2 — surfaced below).
- **(B.2) Caller passes the full `Receiver<I>` keyword as one macro arg.** Splice it directly at the binder position. Zero parser-tokenization conflict.

**Picked (B.2).** Same precedent as `:wat::test::run-hermetic-with-io` (per its DECISION D1 documented at `wat/test.wat:800-815`):

> The aspirational form `(run-hermetic-with-io :i64 :i64 inputs body)` with inner type args ONLY would require constructing `Receiver<i64>` and `Sender<i64>` keywords from the inner type at macro-expand time. No `keyword::from-string` or string->keyword verb exists in the substrate (honest delta D1). Full channel-type keywords are required.

Same substrate property; same honest pivot. The caller spells:

```
(:wat::kernel::run-threads
  :rust::crossbeam_channel::Receiver<wat::core::String>
  :rust::crossbeam_channel::Sender<wat::core::String>
  :my::echo-factory
  :my::echo-client)
```

Verbose-is-honest. The bracket still hides the substrate composition (spawn-thread, peer pair construction, drain-and-join lifecycle); the type-keyword args expose the channel-end vocabulary the wrap-fn needs internally.

### Substrate quirk surfaced (BRIEF STOP trigger 2)

The wat parser tokenizes parametric type keywords (`Receiver<T>`) as a single keyword token. `~` (unquote) inside the `<...>` angle-bracket pair is part of the keyword text, not a quasiquote splice. Evidence: my initial macro form with `[server-rx <- :rust::crossbeam_channel::Receiver<~i-type>]` produced a type-check error with the literal `~i-type` baked into the parametric type: `expected :wat::kernel::ThreadPeer<wat::core::String,wat::core::String>; got :wat::kernel::ThreadPeer<~i-type,~o-type>`. This is consistent with the parser treating `<...>` as part of the keyword lexeme rather than as a structured form open to quasiquote traversal.

**Not a defect; a property.** The fix at the macro layer is to take pre-baked full type keywords as macro args. The fix at the substrate layer (not pursued in D1; documented here for future stones if a real demand surfaces) would be either:
- Make `~` inside parametric `<...>` a structural splice opportunity (parser change), OR
- Expose an `:AST/Keyword` constructor verb at the macro-body level so expand-time keyword construction can build `Receiver<I>` from an `:I` arg, OR
- Implement type-inference on the wrap-fn binders so the macro can leave them un-typed and let the checker fill from the factory's signature (HM-style; substantial work).

None of these are required by D1's BRIEF; the macro-args path is the path the existing `:wat::test::run-hermetic-with-io` macro takes for the same reason.

### Closure-extraction walker (arc 117/133) did NOT fire on the factory adapter

The wrap-fn `(fn [server-rx ...] (factory (ThreadPeer/new server-rx server-tx)))` closes over the `factory` symbol. The closure-extraction walker accepts this — `factory` resolves at expand time to a user-defined fn keyword (e.g. `:my::echo-factory`), and spawn-thread's accept-by-keyword path resolves it through the symbol table at runtime. Zero walker fires; zero diagnostic surfaced. Stone G's walker retirement is unaffected by this Stone.

### `Thread/drain-and-join`'s Result handling

The let-binding `_drained (Thread/drain-and-join thread)` discards the `Result<(), Vec<ThreadDiedError>>` — D1 assumes clean factory exit (per BRIEF § 5 "No panic cascade yet"). The `_` prefix conventionally marks unused bindings; the type checker accepts the discard.

D3 will replace this discard with cascade-policy logic: inspect the Result, if `Err(chain)` propagate as `ProcessGroupErr` and shut down sibling factories; if `Ok(())` keep returning `result` as today. D1's shape is forward-compatible — D3 wraps additional logic around the `_drained` binding without restructuring the let chain.

Same pattern Stone C2's `wat_process_peer_ipc_round_trip.rs` adopted (per its rationale: cleaner than Rust-side AST hand-construction; mirrors how user code will write bracket calls). Zero Rust-side AST building; the test reads as the user would write the bracket call.
